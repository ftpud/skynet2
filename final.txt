The key is already in your code — you stream Claude's response and break early as soon as a valid JSON object is parsed (`parsed_early`). This means you don't need a large `max_tokens` at all, because you stop consuming the stream the moment the JSON is complete.

The problem is that `max_tokens` in Claude controls how many tokens it is *allowed* to generate, not how many you *receive*. If the model hallucinates/repeats, it burns through those tokens server-side and you still wait.

Here are your options, from best to worst:

---

**1. BEST — Add `stop_sequences` (already supported by Claude API)**
Tell Claude to stop generating after the closing brace of the JSON. Add this to your `client.messages.stream(...)` call:

python
stop_sequences=["}\n", "}\n\n"],


But a smarter stop is to detect the JSON is complete and close the stream yourself — which you already do with `break`. The issue is the SDK may still buffer/wait. Force-close the stream:

python
if parsed_early is not None:
    stream.close()   # <-- force close the HTTP connection immediately
    break


**2. BETTER — Use a small `max_tokens` + retry on truncation**
For a single JSON action response, Claude never needs more than ~500 tokens. Set:
yaml
max_tokens: 512

If the response is truncated (stop_reason == 'max_tokens'), retry. This hard-caps server-side generation time.

**3. CURRENT CODE FIX — Force-close the stream on early parse**
In your Claude streaming block, change:
python
if parsed_early is not None:
    # ... usage extraction ...
    break

to:
python
if parsed_early is not None:
    try:
        stream.close()  # kill the HTTP stream immediately
    except Exception:
        pass
    break


---

**Summary of recommended changes:**

1. In `agents/*.yaml` for Claude agents, set `max_tokens: 512` (or up to 1024 for complex answers)
2. Call `stream.close()` before `break` in the Claude streaming block to immediately drop the connection when JSON is parsed
3. Optionally add `stop_sequences=["}\n"]` to the stream call as a server-side hint
